# 产品需求文档 (PRD)：个人量化系统 (QuantDash)

## 1. 项目背景与愿景

### 1.1 项目背景

QuantDash 当前已经具备个人量化看板雏形：自选股、实时行情、K 线、市场指数、新闻/舆情占位、财务估值占位。这个阶段能帮助用户“看见市场”，但还不能支撑完整量化流程。

一个完备的个人量化系统不只是行情展示，而是需要形成闭环：

```text
数据获取
  -> 数据清洗
  -> 股票池构建
  -> 因子计算
  -> 策略研究
  -> 回测验证
  -> 组合构建
  -> 风险控制
  -> 模拟交易
  -> 实盘执行
  -> 监控复盘
```

### 1.2 项目目标

为量化领域新手、但具备全栈开发能力的个人投资者，构建一个从行情观察到策略验证、再到模拟交易和半自动实盘的个人量化工作台。

系统目标分三层：

- **第一层：看清市场**：行情、K 线、新闻、公告、舆情、财务指标。
- **第二层：验证想法**：数据中台、股票池、因子、策略、回测、报告。
- **第三层：谨慎执行**：组合、风控、模拟交易、半自动实盘、复盘归因。

### 1.3 核心价值

- 帮助新手用结构化方式学习量化，而不是直接追逐复杂模型。
- 把“投资想法”转化为可以验证的策略规则。
- 通过可信回测识别无效策略，避免凭感觉实盘。
- 通过风控、模拟盘和复盘降低实盘风险。

### 1.4 重要原则

- **不优先做全自动实盘**：先做数据、回测、模拟盘，再考虑实盘。
- **不只看收益率**：必须同时关注最大回撤、交易成本、胜率、换手率、超额收益。
- **不做黑箱策略起步**：新手阶段优先做可解释、可复盘的日频策略。
- **不相信未经清洗的数据**：量化系统的地基是数据质量。

## 2. 用户画像与使用场景

### 2.1 目标用户

- 个人投资者。
- 具备全栈开发背景。
- 量化领域新手。
- 希望从看行情逐步进阶到策略研究、回测和模拟交易。

### 2.2 典型使用场景

| 场景 | 用户问题 | 系统能力 |
| --- | --- | --- |
| 市场观察 | 今天市场发生了什么？ | 行情、指数、新闻、公告、舆情 |
| 标的筛选 | 哪些股票值得研究？ | 股票池、过滤条件、因子排名 |
| 策略构思 | 我的想法能不能规则化？ | 策略模板、参数配置、信号生成 |
| 历史验证 | 这个策略过去有效吗？ | 回测引擎、风险指标、交易明细 |
| 组合管理 | 应该买多少？ | 组合构建、仓位约束、调仓计划 |
| 风险控制 | 最坏可能亏多少？ | 最大回撤、仓位限制、流动性限制 |
| 实盘前验证 | 真实行情下能不能跑？ | 模拟交易、虚拟持仓、模拟成交 |
| 每日复盘 | 今天为什么赚/亏？ | 收益归因、持仓复盘、交易日志 |

## 3. 产品模块总览

QuantDash 从看板升级为完整量化系统后，分为以下核心模块：

```text
QuantDash
├── 1. Market Dashboard 行情看板
├── 2. Data Center 数据中心
├── 3. Universe Center 股票池中心
├── 4. Factor Engine 因子引擎
├── 5. Research Lab 策略研究
├── 6. Backtest Center 回测中心
├── 7. Portfolio Center 组合中心
├── 8. Risk Center 风控中心
├── 9. Paper Trading 模拟交易
├── 10. Live Trading 实盘交易
└── 11. Monitoring & Review 监控复盘
```

每个模块都需要尽量保持独立，能够单独开发、单独测试，并通过清晰接口协作。

## 4. 核心功能模块

### 4.1 Market Dashboard 行情看板

#### 4.1.1 模块定位

行情看板负责让用户快速理解市场状态，是当前系统的起点，但不是完整量化系统的终点。

#### 4.1.2 功能需求

- 股票自选池：
  - 支持 A 股 6 位代码搜索。
  - 支持名称、拼音首字母搜索。
  - 支持添加、删除、选中股票。
  - MVP 阶段使用 `localStorage`，后续迁移到 PostgreSQL。
- 实时行情：
  - 展示现价、涨跌幅、涨跌额、成交量、成交额、最高价、最低价。
  - 自选股 quote 按固定频率轮询。
- 市场指数：
  - 展示上证指数、深证成指、创业板指、科创 50。
- K 线与技术图表：
  - 支持日线、周线、月线、年线。
  - 支持 MA5、MA10、成交量。
  - 后续支持分时图、量比、盘口五档、逐笔成交。
- 新闻与公告：
  - 展示个股相关新闻。
  - 展示交易所公告、财报公告、回购、减持、增持等事件。
- 舆情监控：
  - 聚合微博、雪球、抖音等财经热点关键词。
  - 展示全市场热词和个股相关热度。
- 财务卡片：
  - 展示 PE TTM、PB、ROE、毛利率等指标。
  - 后续展示估值水位。

#### 4.1.3 核心接口

```text
GET /api/market/indexes
GET /api/stock/quote/[symbol]
GET /api/stock/kline/[symbol]?type=daily
GET /api/stock/intraday/[symbol]
GET /api/stock/order-book/[symbol]
GET /api/stock/trades/[symbol]
GET /api/stock/news/[symbol]
GET /api/stock/announcements/[symbol]
GET /api/intelligence/hot-keywords
```

#### 4.1.4 验收标准

- 用户能添加一只股票并看到真实 quote。
- 用户能切换日/周/月/年 K 线。
- 行情源异常时页面不白屏，展示降级状态。
- 新闻、舆情、财务数据缺失时展示空状态，而不是伪造结论。

### 4.2 Data Center 数据中心

#### 4.2.1 模块定位

数据中心是整个量化系统的地基。行情展示可以容忍短暂缺失，但回测和策略研究不能使用脏数据。

#### 4.2.2 需要支持的数据类型

| 数据类型 | 用途 |
| --- | --- |
| 股票基础信息 | 名称、代码、交易所、行业、上市日期 |
| 交易日历 | 判断调仓日、节假日、停牌日 |
| 日线行情 | 日频策略、收益计算、K 线展示 |
| 分钟线行情 | 分时分析、分钟级策略扩展 |
| Tick 数据 | 高频扩展，不作为新手阶段优先项 |
| 前复权/后复权价格 | 避免分红送股造成历史价格失真 |
| 涨跌停价格 | A 股回测必须考虑买不进/卖不出 |
| 停牌数据 | 避免回测假设停牌股票可交易 |
| ST 状态 | 股票池过滤 |
| 财务数据 | 价值、质量、成长因子 |
| 指数成分股 | 构建沪深 300、中证 500 等股票池 |
| 行业分类 | 行业暴露和行业中性 |
| 新闻公告 | 事件驱动策略、风险提示 |
| 舆情数据 | 情绪因子、热点观察 |

#### 4.2.3 数据分层

```text
Raw 原始层
  保存外部源原始响应，方便排查问题

Clean 清洗层
  字段标准化、类型转换、去重、异常值处理

Adjusted 复权层
  生成前复权/后复权价格序列

Feature 特征层
  生成因子、标签、策略可读特征

Serving 服务层
  提供前端、回测、策略读取接口
```

#### 4.2.4 功能需求

- 数据源适配：
  - MVP 可使用 Sina 公开行情接口。
  - 研究与回测优先接入 AkShare。
  - 后续可扩展 Tushare、东方财富、JoinQuant、RiceQuant。
- 数据质量检查：
  - 缺失值检查。
  - 重复交易日检查。
  - 异常涨跌幅检查。
  - 成交量/成交额异常检查。
- 数据更新任务：
  - 日线行情每日收盘后更新。
  - 股票基础信息每日更新。
  - 财务数据按财报周期更新。
  - 新闻舆情按分钟或小时更新。
- 数据可追溯：
  - 保留来源字段 `source`。
  - 保留更新时间 `updated_at`。
  - 关键数据保留原始 payload。

#### 4.2.5 验收标准

- 可以查询任意 A 股最近 10 年日线数据。
- 可以区分未复权、前复权、后复权价格。
- 可以判断某只股票某日是否停牌、是否 ST、是否涨跌停。
- 每个数据接口返回 `source` 和 `updated_at`。

### 4.3 Universe Center 股票池中心

#### 4.3.1 模块定位

股票池中心回答一个核心问题：策略今天允许交易哪些股票。

新手不应该直接对全市场乱跑策略，应先定义可控股票池，再基于过滤条件筛选。

#### 4.3.2 功能需求

- 内置股票池：
  - 全 A。
  - 沪深 300。
  - 中证 500。
  - 中证 1000。
  - 创业板。
  - 科创板。
- 过滤条件：
  - 剔除 ST。
  - 剔除停牌。
  - 剔除上市不足 N 天。
  - 剔除成交额过低。
  - 剔除价格过低或过高。
  - 剔除财务异常股票。
  - 剔除当日涨停不可买股票。
  - 剔除当日跌停不可卖股票。
- 股票池版本管理：
  - 保存股票池配置。
  - 保存每日股票池成分。
  - 支持回测按历史日期读取股票池，避免未来函数。

#### 4.3.3 核心接口

```text
GET /api/universes
POST /api/universes
GET /api/universes/[id]/stocks?date=YYYY-MM-DD
POST /api/universes/[id]/preview
```

#### 4.3.4 验收标准

- 用户可以创建一个“沪深 300 + 剔除 ST + 剔除停牌”的股票池。
- 回测可以按历史日期读取当日股票池。
- 股票池结果可以解释每只股票被剔除的原因。

### 4.4 Factor Engine 因子引擎

#### 4.4.1 模块定位

因子是量化策略的原材料。一个因子可以理解为“给股票打分的指标”。

#### 4.4.2 新手优先支持的因子

| 因子类型 | 示例 |
| --- | --- |
| 动量因子 | 20 日涨幅、60 日涨幅、120 日涨幅 |
| 反转因子 | 1 日跌幅、5 日跌幅 |
| 价值因子 | PE TTM、PB、股息率 |
| 质量因子 | ROE、毛利率、资产负债率 |
| 成长因子 | 营收增长率、净利润增长率 |
| 波动因子 | 20 日波动率、最大回撤 |
| 流动性因子 | 成交额、换手率、成交额均值 |
| 情绪因子 | 舆情热度、新闻数量、热度变化 |

#### 4.4.3 功能需求

- 因子定义：
  - 使用 Python 函数或配置定义因子。
  - 每个因子必须包含名称、输入数据、计算逻辑、更新频率。
- 因子计算：
  - 支持按日期批量计算。
  - 支持按股票池计算。
  - 支持增量更新。
- 因子存储：
  - 保存 `date`, `symbol`, `factor_name`, `value`。
  - 保存因子版本，方便回测复现。
- 因子分析：
  - 分组收益。
  - IC / Rank IC。
  - 因子覆盖率。
  - 因子分布。

#### 4.4.4 核心接口

```text
GET /api/factors
POST /api/factors/run
GET /api/factors/[name]/values?date=YYYY-MM-DD
GET /api/factors/[name]/analysis
```

#### 4.4.5 验收标准

- 可以计算某日全市场 20 日动量因子。
- 可以查看某个因子的覆盖率和分布。
- 策略回测可以读取历史因子值，而不是实时重新计算导致不可复现。

### 4.5 Research Lab 策略研究

#### 4.5.1 模块定位

策略研究模块负责把投资想法变成明确、可执行、可回测的规则。

一个策略必须回答：

```text
买什么？
什么时候买？
买多少？
什么时候卖？
如何控制风险？
多久调仓？
```

#### 4.5.2 新手策略模板

第一阶段优先提供简单、可解释的日频策略模板：

- 动量策略：
  - 股票池：沪深 300 / 中证 500。
  - 买入：过去 20 日涨幅排名前 N。
  - 过滤：成交额、ST、停牌、涨跌停。
  - 持仓：等权。
  - 调仓：每周或每月。
- 价值质量策略：
  - 买入：低 PB + 高 ROE。
  - 过滤：财务异常、流动性不足。
  - 持仓：等权或因子分数加权。
- 情绪事件策略：
  - 买入候选：舆情热度上升但未明显大涨的股票。
  - 风控：严格限制单票仓位和持有天数。

#### 4.5.3 功能需求

- 策略配置：
  - 股票池。
  - 因子。
  - 排名规则。
  - 过滤规则。
  - 调仓频率。
  - 持仓数量。
  - 单票上限。
  - 止损/止盈规则。
- 策略代码：
  - 支持 Python 策略文件。
  - 每个策略实现统一接口。
- 策略版本：
  - 每次回测绑定策略版本和参数。
  - 防止后续改代码导致历史回测不可复现。

#### 4.5.4 策略接口草案

```python
def generate_signals(context) -> list[Signal]:
    """
    输入：历史行情、股票池、因子、当前持仓、当前日期
    输出：买入/卖出/调仓信号
    """
```

#### 4.5.5 验收标准

- 用户可以配置一个“20 日动量 + 每周调仓 + 等权 10 只”的策略。
- 策略可以生成每日目标持仓或交易信号。
- 策略参数变更后，历史回测能记录对应版本。

### 4.6 Backtest Center 回测中心

#### 4.6.1 模块定位

回测中心是完整量化系统的核心。它负责验证策略过去是否有效，并尽量模拟真实交易约束。

#### 4.6.2 必须避免的新手陷阱

| 陷阱 | 说明 |
| --- | --- |
| 未来函数 | 使用了当时不可能知道的数据 |
| 幸存者偏差 | 只使用现在还存在的股票 |
| 忽略手续费 | 收益虚高 |
| 忽略滑点 | 成交价格过于理想 |
| 忽略涨跌停 | A 股中买不进/卖不出很常见 |
| 忽略停牌 | 停牌股票现实中无法交易 |
| 忽略 T+1 | A 股当天买入不能当天卖出 |
| 只看收益率 | 忽略回撤、波动、换手和风险 |

#### 4.6.3 功能需求

- 回测配置：
  - 策略。
  - 股票池。
  - 起止日期。
  - 初始资金。
  - 手续费。
  - 滑点。
  - 调仓频率。
  - 基准指数。
  - 单票仓位上限。
  - 最大持仓数量。
- 交易约束：
  - T+1。
  - 停牌不可交易。
  - 涨停不可买。
  - 跌停不可卖。
  - 成交额不足限制下单规模。
- 回测结果：
  - 净值曲线。
  - 回撤曲线。
  - 年化收益。
  - 最大回撤。
  - 夏普比率。
  - Calmar 比率。
  - 胜率。
  - 换手率。
  - 超额收益。
  - 每日持仓。
  - 每笔交易。

#### 4.6.4 核心接口

```text
POST /api/backtests
GET /api/backtests
GET /api/backtests/[id]
GET /api/backtests/[id]/equity-curve
GET /api/backtests/[id]/drawdown
GET /api/backtests/[id]/positions
GET /api/backtests/[id]/trades
```

#### 4.6.5 验收标准

- 可以完成一个日频策略的 3 年历史回测。
- 回测报告展示收益、回撤、交易明细和每日持仓。
- 回测中停牌、涨跌停、手续费、滑点可配置。
- 回测结果可复现：同策略、同参数、同数据版本输出一致。

### 4.7 Portfolio Center 组合中心

#### 4.7.1 模块定位

策略给出“买哪些股票”，组合中心负责决定“每只买多少”。

#### 4.7.2 功能需求

- 组合构建方式：
  - 等权重。
  - 按因子分数加权。
  - 按波动率倒数加权。
  - 行业中性加权。
- 仓位约束：
  - 单票最大仓位。
  - 行业最大仓位。
  - 总仓位上限。
  - 现金保留比例。
- 调仓计划：
  - 从当前持仓计算目标持仓。
  - 从目标持仓生成买卖订单。
  - 展示调仓差异。

#### 4.7.3 新手默认方案

第一阶段使用最容易理解的等权组合：

```text
选出 10 只股票
每只目标仓位 10%
单票上限 10%
总仓位上限 95%
保留 5% 现金
```

#### 4.7.4 核心接口

```text
POST /api/portfolios/target
GET /api/portfolios/[id]/positions
GET /api/portfolios/[id]/rebalance-plan
```

#### 4.7.5 验收标准

- 输入一组策略信号后，系统能生成目标持仓。
- 当前持仓和目标持仓不同，系统能生成调仓计划。
- 单票、行业、总仓位约束能够生效。

### 4.8 Risk Center 风控中心

#### 4.8.1 模块定位

风控中心负责防止策略在异常行情、异常数据或过高仓位下造成不可接受损失。

#### 4.8.2 风控层级

| 层级 | 示例 |
| --- | --- |
| 策略级风控 | 最大回撤超过阈值暂停策略 |
| 组合级风控 | 单票不超过 10%，行业不超过 30% |
| 交易级风控 | 涨停不追买，流动性不足不下单 |
| 数据级风控 | 数据缺失、价格异常时停止交易 |

#### 4.8.3 功能需求

- 最大回撤监控。
- 单票仓位限制。
- 行业仓位限制。
- 总仓位限制。
- 流动性限制。
- 涨跌停交易限制。
- 停牌交易限制。
- 异常数据拦截。
- 连续亏损降仓。

#### 4.8.4 核心接口

```text
POST /api/risk/check-order
POST /api/risk/check-portfolio
GET /api/risk/limits
PATCH /api/risk/limits
```

#### 4.8.5 验收标准

- 风控可以拦截超过单票仓位上限的订单。
- 风控可以拦截停牌、涨停买入、跌停卖出。
- 数据异常时，系统能阻止策略继续下单。

---

### 4.9 Paper Trading 模拟交易

#### 4.9.1 模块定位

模拟交易是实盘前必须经过的阶段。它使用真实行情和虚拟资金，验证策略在实时环境中的表现。

#### 4.9.2 功能需求

- 虚拟账户：
  - 初始资金。
  - 可用现金。
  - 当前持仓。
  - 总资产。
  - 每日净值。
- 模拟订单：
  - 买入。
  - 卖出。
  - 撤单。
  - 部分成交。
- 模拟成交：
  - 使用实时 quote 或下一个 bar 模拟成交。
  - 扣除手续费和滑点。
- 每日记录：
  - 交易明细。
  - 持仓明细。
  - 净值变化。
  - 策略日志。

#### 4.9.3 核心接口

```text
POST /api/paper/accounts
GET /api/paper/accounts/[id]
POST /api/paper/orders
GET /api/paper/orders
GET /api/paper/positions
GET /api/paper/equity-curve
```

#### 4.9.4 验收标准

- 策略可以每天生成模拟调仓订单。
- 模拟账户能正确更新现金、持仓和净值。
- 模拟盘至少连续运行 1-3 个月后，再考虑实盘。

---

### 4.10 Live Trading 实盘交易

#### 4.10.1 模块定位

实盘交易是最后阶段。对新手而言，第一版实盘应优先做“半自动交易”，即系统生成建议订单，用户人工确认后执行。

#### 4.10.2 功能需求

- 券商接口适配。
- 账户资金同步。
- 持仓同步。
- 订单创建。
- 撤单。
- 成交回报。
- 失败重试。
- 交易审计日志。
- 人工确认机制。
- 紧急停止按钮。

#### 4.10.3 风险原则

- 默认不启用全自动下单。
- 实盘前必须通过模拟交易验证。
- 实盘必须有单日亏损限制。
- 实盘必须有最大下单金额限制。
- 实盘必须记录完整操作日志。

#### 4.10.4 核心接口

```text
GET /api/live/accounts
GET /api/live/positions
POST /api/live/orders/preview
POST /api/live/orders/confirm
POST /api/live/orders/cancel
GET /api/live/executions
POST /api/live/emergency-stop
```

#### 4.10.5 验收标准

- 实盘订单下发前必须经过风控检查。
- 实盘订单下发前必须人工确认。
- 成交回报、失败、撤单都必须可追溯。

---

### 4.11 Monitoring & Review 监控复盘

#### 4.11.1 模块定位

监控复盘负责回答：今天为什么赚钱或亏钱，策略是否还值得继续运行。

#### 4.11.2 功能需求

- 策略运行监控：
  - 是否按时生成信号。
  - 是否按时完成调仓。
  - 数据源是否正常。
  - 订单是否异常。
- 业绩复盘：
  - 今日收益。
  - 累计收益。
  - 相对基准收益。
  - 最大回撤。
  - 当前持仓。
  - 今日交易。
  - 行业暴露。
  - 风险事件。
- 归因分析：
  - 收益来自哪些股票。
  - 亏损来自哪些股票。
  - 行业贡献。
  - 风格因子贡献。
  - 交易成本影响。
- 告警：
  - 数据源异常。
  - 策略运行失败。
  - 回撤超过阈值。
  - 订单失败。
  - 舆情或公告重大事件。

#### 4.11.3 核心接口

```text
GET /api/monitoring/health
GET /api/reports/daily
GET /api/reports/performance
GET /api/reports/attribution
POST /api/alerts
```

#### 4.11.4 验收标准

- 每日能生成一份策略日报。
- 用户能看到收益、回撤、持仓、交易、异常。
- 系统异常能通过页面或消息渠道提醒用户。

## 5. 技术架构

### 5.1 前端

- **框架**：Next.js App Router。
- **语言**：TypeScript。
- **状态管理**：Zustand。
- **样式**：Tailwind CSS + Shadcn UI。
- **图表**：Apache ECharts。
- **页面模块**：
  - 行情看板。
  - 数据中心。
  - 股票池中心。
  - 因子中心。
  - 策略研究。
  - 回测报告。
  - 组合管理。
  - 风控中心。
  - 模拟交易。
  - 监控复盘。

### 5.2 BFF 层

- 使用 Next.js Route Handlers。
- 负责前端请求聚合、鉴权、参数校验和后端转发。
- 前端不直接访问 Python 服务和外部数据源。

### 5.3 后端数据服务

- **框架**：Python FastAPI。
- **数据处理**：Pandas / Polars。
- **数据源**：Sina、AkShare，后续可扩展 Tushare、东方财富等。
- **任务调度**：APScheduler / Celery / Prefect。
- **回测引擎**：自研日频事件驱动回测，后续可评估 Backtrader、Qlib 等框架。

### 5.4 存储

| 存储 | 用途 |
| --- | --- |
| PostgreSQL | 用户、自选股、股票基础信息、财务、新闻、回测记录 |
| Redis | 高频 quote、盘口、分时、任务状态缓存 |
| Parquet / DuckDB | 大规模历史行情、因子矩阵、回测读取 |
| localStorage | MVP 阶段自选股本地保存 |

## 9. 新手学习与实现顺序

作为量化新手，推荐按照以下顺序建设：

```text
1. 行情看板
2. 数据中心
3. 股票池
4. 基础因子
5. 第一个简单策略
6. 日频回测
7. 回测报告
8. 组合和风控
9. 模拟交易
10. 半自动实盘
```

最重要的阶段不是实盘，而是：

- 数据是否正确。
- 回测是否可信。
- 策略是否可解释。
- 风控是否明确。
- 模拟盘是否稳定。

## 10. 非目标

以下内容不作为早期目标：

- 高频交易。
- 复杂机器学习选股。
- 全自动实盘下单。
- 期货、期权、外汇、加密货币多市场扩展。
- 社交化跟单。
- 面向多用户商业 SaaS 的权限体系。

这些能力只有在日频股票策略、回测、风控、模拟盘稳定后再考虑。
